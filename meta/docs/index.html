Purpose

The purpose of the website actually is twofold. First, it serves as a digital personal identity or reference. Simply speaking, it's a private homepage. Additionally, it serves as a content publication and distribution platform. It contains my blog, further documents and papers, slides and downloads. Well, that's all about it.

This documentation mainly takes care of the website implementation. Although there's some strategic / design documentation as well, the vast amount of the documentation will be about the technical concepts and aspects of the website.

Overview

Topics

The website will mainly be about my personal interests - no more no less. However, the main topics of this website will be about my professional life and experience. More precisely, computing, organisational development, team leadership and software engineering will be the main areas.

Types of content

The website will mainly serve three different types of content. The first type is just plain document-centric pages with individual topics and length. Generally speaking, this type of content is called article.

The second category of content is a typical index-oriented page. Its purpose is to provide a list of references to other pages, typically meaningfully grouped and sorted. The index class of pages is called index.

The third class of content actually is a presentational page. Typically, this type of page has much more visual weight. Even different types of media, such as videos or animations are possible. This third page category's name is slide.

Site structure

Conceptually, the site is divided into separate areas. The current implementation hosts the following areas:

  • Main
    This is the main area where the master index (home page) and the articles reside. Despite supportive pages such as about, imprint and others, this area hosts all blog posts.
  • Talks
    Obviously, this area is about public speaking and presentation. Here presentations, slides and other material (maybe sample code) are provided.
  • Meta
    This area hosts self-descriptive content only. It contains this documentation, technical pages (such as reference pages and test pages) as well as reports.

Technology

There's nothing really special to this website, actually. The basic content types being used are:

  • HTML: For template layouts. This is obvious. It's a website after all.
  • LESS: For stylesheet definitions.
  • COFFEE: For javascripts.
  • MARKDOWN: For all textual content and documentation.

As for the toolchain, it's mostly noteworthy that large parts of the website are basically made up by static pages and resources. The static pages are generated from the source content using hyde, a so called static website generator written in python. That being said, it's clear that the overall development has python as preferred language and platform.

In the context of hyde, the page templates are created using the powerful jinja2 template engine. The stylesheet files are compiled using less.js, the javascript files obviously compiled through coffeescript compiler. Both less and coffee require a javascript runtime, for which node.js is being used.

Concepts

Organisation

Device Types

The layout concept basically will support 3 layouts depending on the target device. Those are:

  • Screen: The default screen layout, known as the "classic" desktop view.
  • Mobile: The layout for small screen devices, such as smartphones.
  • Print: The layout for printers.

Device targetting is done solely via CSS stylesheets. At the moment, the implementation strategy does not use media queries. Instead, a typical query on user-agent HTTP request header is performed to recognize device type. For all devices, the markup remains stable while styles and javascripts may differ.

Currently, only the screen and mobile device types (views) are implemented. The print view will eventually added in future.

Page Types

The [layout reference] deals with a number of so called "page types", which essentially categorize content into separate frequently used representations. The page types discussed and supported herein are:

  • Article: A typical, textual article / blog page.
  • Listing: A typical, mostly textual listing page.
  • Home: The primary index overview page (home page).

Apart from these core page types, several other page types are being recognized. However, their representation is not yet documented. Some of the (yet only informally existing) page types are:

  • Slides: Typical slide presentations within the site.
  • Talks: An overview for the abovementioned slide presentations.
  • Bio: A strong visual presentation of personal experience and capabilities.
  • History: A typical archive / timeline / journal presentation of content.
Article Page Type

This [page type] is most probably the most important page type of the entire website. It's used for any textual content which can be classified as blog post or standalone article.

Example: The article layout reference at refs/article.html

It mostly takes all features provided by the master layout, plus some minor additions like a related articles footer.

Home Page Type

The home page is te first page to be displayed to the user in desktop mode. It should contain a very easy and quick understandable navigation and provide a fast overview of what ilker.de has to offer.

Example: refs/home.html

Second, the home page actually should transport the primary purpose of the site - the "digital identity" of me. That actually means that information about me as a person or public individual should be available at first sight - or just a single click away.

Hence, the splash page will contain:

  • The main navigation (index, articles, talks, about)
  • The latest article (as title)
  • An "editors pick" of an article (should differ from latest article, updated quarterly)
  • Information about me as a person and the services I can deliver (portfolio, cv)
  • Contact Info (email, twitter, xing, linkedin)

Hyde

Layouts

Adhering to the original hyde directory structure, all layout templates can be found in the layouts folder. Templates are written with jinja, of course.

The general concept of the templates is quite straight-forward. All templates correspond to a general [Page Type], except one single template called the Master. The master template (file: master.j2) is the common style and layout definition of the website, as translated from the [Master Website]. All other templates inherit from master, but never from themselves. More strictly speaking: there's only a single hierarchy - no more.

All templates may use visualization helpers, which are called macros. They are organized within the macro folder and contain commonly used functionality, such as translations or navigation.

Variables

Templates make heavy use of the hyde ecosystem, especially the resource element. For commonly used stuff, such as meta or the [Translation Macro], the master template defines single-letter top-level variables.

Source Code

The repository for this website has a very specific branching model. The branches used are:

  • default: The main branch. The mainline represents release status. It typically is in sync with the below mentioned publish branch.
  • content: A branch holding the content of the website, alonside with the code. Used for writing articles and other content. Represents draft status.
  • dev: The development branch, without real content. That's the bare website 'application'. Represents dev status.
  • publish: A fork of the development branch. Used to publish the code part of the website to public bitbucket / github repositories.

Specialities

Commenting

Since turning off website comments completely, a number of responses through various other channels were made regarding this topic. While a few responses were positive, others think a little different. It seems as if the motivation to comment and discuss a specific topic on the website went down significantly. This case addresses this issue with appropiate counter-measurements without actually providing on-site commenting feature.

The preferred solution is to provide a 'write comment' button which actually links to a separate, descriptive comments page. On that particular page, the user can find a short notice about the 'comments policy' of the website, links to further explanations regarding the 'no on-site comments policy' and at least a very prominent area with further options.

The aforementioned options are

  • a link launching the users MUA with prefilled mail and subject fields,
  • a link enabling to share the page url in question with twitter,
  • a link enabling to share the page url in question with google plus,

whereas the mailto: link is required to be provided. The social sharing links are optional only.

The comments page will be a single page for all pages motivating to write comments. In plain english, there's only one comments page for all other pages. To customize the comments page visualization and content towards the specific page in question, the comments page can receive parameters within its URL, such as subject or referrer.

The provided data ideally should be processed on the client-side using Javascript. In consequence, at least the 'write comment' button on the comments page can't be used if JS is disabled. That's fair enough since nowadays mostly only crawlers don't execute JS. Despite that, it's obvious that no server resources are consumed for this client-specific operation.
